SYS - 01

The whole loop, on one sheet

tinyux keeps the data path intentionally boring: browser events, optional survey prompts, one collector endpoint, and tables you own.

01 / embed
One static file

No framework, package manager, extension, or vendor account required.

02 / observe
Behavioral signals

Page views, clicks, aggregate heatmap points, scroll depth, focus changes, section views, and heartbeats.

03 / ask
Micro-surveys

Rating, yes/no, choices, and optional text prompts with cooldown rules.

04 / report
Batched delivery

sendBeacon and keepalive fetch, plus local retry for failed sends.

05 / own
Your database

PHP/SQLite collector stores clients, sessions, events, surveys, and failures.

Add tinyux UX telemetry to my project:
1. Fetch https://tanzir71.github.io/tinyuxjs/tinyux.js into my static assets.
2. Add the script tag to key product pages with data-api-url pointing at /collect.php.

4. Add one click survey and one section-scroll survey.

The machine-readable guide lives at /llms.txt.

One command

Grab the files

Download the client and collector pieces, then initialize SQLite with the schema.

# client library
curl -LO https://tanzir71.github.io/tinyuxjs/tinyux.js

# collector schema
curl -LO https://tanzir71.github.io/tinyuxjs/schema.sql
sqlite3 tinyux.db < schema.sql

Put `collect.php`, `tinyux.js`, and `tinyux.db` where PHP can read/write them.

Manual

Copy the tag

Already have the file hosted? Drop the tag and add triggers where context matters.

<script src="/js/tinyux.js"
  data-app-id="acme-app"
  data-api-url="/collect.php"
  data-config='{"surveys":{"maxPerSession":2}}'>
</script>

See declarative triggers for click, section, time, and exit-intent surveys.

SYS - 03

Signals small enough to trust

tinyux records metadata about interaction context. It can collect aggregate click/tap heatmap points, but it does not replay a user's screen, record typed content, or ship private product data to a third-party workspace.

Page views

Initial URL, referrer, viewport, scroll depth, focus time, and client/session IDs.

page_view

Interactions

Bind selectors or add attributes to key CTAs, forms, and controls.

interaction

Section views

IntersectionObserver-powered section visibility, with cooldown and once-per-session options.

section_view

Focus and blur

Know whether a user left the page during a task or returned to continue.

focus - blur

Aggregate heatmaps

Optional click/tap density points with coarse coordinates and sanitized element metadata.

heatmap_point

User association

Attach your own user, account, plan, or role IDs when your app already knows them.

identify

Corner prompts

Use blocking modal surveys or subtle non-blocking prompts in any corner.

ui.mode

Exit intent

Detect top-edge pointer movement and ask one last contextual question.

exit_intent

Heartbeats

Periodic pulses make abandoned sessions and long-running tasks visible server-side.

heartbeat

Input metadata

Optional lengths/types only. Values are never captured by default.

input_meta

Survey lifecycle

Shown, closed, skipped, and submitted events tie answers to page context.

survey_shown - survey_submitted
SYS - 04

Why use tinyux instead of a behavior suite?

Hotjar, FullStory, LogRocket, and PostHog are far larger systems. tinyux is for the narrower job: collect useful UX context without buying a platform.

Self-hosted by default

Your events and survey responses land in your own SQLite database. No replay vendor owns the raw trail.

Friction at the source

Ask right after a confusing click, a pricing section view, or an exit intent moment.

Small enough to audit

One readable client file and one PHP collector. No black-box recorder and no account setup.

SYS - 05

What it deliberately does not do

Honest limits keep tinyux useful. If you need full product analytics, choose a real analytics platform. If you need session replay, choose a replay tool.

No replay theater

No DOM playback, video-style recordings, screenshots, console logs, network capture, or keystroke contents. Heatmaps are aggregate click/tap coordinates only.

No false certainty

Client-side telemetry is directional. Use it to find friction, prompt follow-up, and prioritize fixes, not to claim perfect behavioral truth.

support - open source

Small tools survive when people use them.

tinyux is MIT-licensed and built for teams that need an honest feedback layer without a procurement loop. Issues, stars, and coffee all help.
