Implementation guide

Everything you need to add lightweight UX telemetry to a product page: one browser script, declarative triggers, contextual surveys, optional aggregate heatmaps, and one collector endpoint you own.

Quick start

Host `tinyux.js` on your own domain and point it at a collector endpoint. The client starts automatically when both `data-app-id` and `data-api-url` are present.

product-page.html
<script src="/js/tinyux.js"
  data-app-id="acme-app"
  data-api-url="/collect.php"
  data-config='{
    "surveys": {"maxPerSession": 2, "minIntervalMinutes": 5},
    "exitIntent": {"enabled": true, "survey_id": "exit-survey"}
  }'>
</script>

Create the SQLite database with:

sqlite3 tinyux.db < schema.sql

Declarative triggers

Add attributes to product UI when you want telemetry or a survey without writing custom JavaScript.

TriggerAttributeBehavior
Click`data-tinyux-trigger="click"`Captures element metadata and optionally shows `data-tinyux-survey-id`.
Section scroll`data-tinyux-trigger="section-scroll"`Records section visibility with threshold, cooldown, and once-per-session controls.
Time delay`data-tinyux-trigger="time"`Shows a survey after `data-tinyux-after-ms`, defaulting to 5000 ms.
Exit intent`data-tinyux-trigger="exit-intent"`Associates a survey with global exit-intent detection.
 
<button
  data-plan="team"
  data-tinyux-trigger="click"
  data-tinyux-survey-id="upgrade-clarity">
  Upgrade
</button>

<section
  id="pricing"
  data-tinyux-trigger="section-scroll"
  data-tinyux-threshold="0.5"
  data-tinyux-once="session"
  data-tinyux-survey-id="pricing-clarity">
</section>

Surveys

Surveys can be registered programmatically or embedded as JSON in `data-tinyux-survey`. Supported types are `rating`, `yesno`, `choice`, and `text`.

const ux = window.tinyux.getInstance()

ux.registerSurvey({
  id: 'pricing-clarity',
  type: 'yesno',
  title: 'Pricing clarity',
  question: 'Was this section clear?',
  oncePerSession: true,
  cooldownMinutes: 120,
  ui: { mode: 'corner', position: 'bottom-right' }
})

ux.registerSurvey({
  id: 'exit-survey',
  type: 'choice',
  title: 'Before you go',
  question: 'What stopped you today?',
  choices: ['Pricing', 'Missing feature', 'Just browsing', 'Other']
})

Questions can interpolate element dataset values with `{{data-name}}`, using safe dataset metadata only.

Survey UI defaults to a blocking center modal. Set `ui.mode` to `corner` for a non-blocking prompt, with `position` set to `bottom-right`, `bottom-left`, `top-right`, or `top-left`.

JavaScript API

`tinyux.destroy()`Stop heartbeat/bindings and clear the singleton.

Identity association

tinyux keeps anonymous `client_id` and `session_id` by default. If your app already knows the signed-in user or account, pass that identity explicitly. tinyux does not fingerprint browsers or devices to guess identity.

const ux = window.tinyux.getInstance()

ux.identify('user_123', {
  account_id: 'acct_456',
  plan: 'team',
  role: 'admin'
})

ux.track('checkout_started')

You can also provide the initial user in config:

tinyux.init({
  appId: 'acme-app',
  apiUrl: '/collect.php',
  user: {
    id: 'user_123',
    account_id: 'acct_456',
    traits: { plan: 'team' }
  }
})

Aggregate heatmaps

Heatmaps are opt-in and aggregate-only. tinyux records coarse click/tap coordinates as `heatmap_point` events so you can build a simple density overlay or export later. It does not capture screenshots, DOM snapshots, mouse trails, or replayable sessions.

<script src="/js/tinyux.js"
  data-app-id="acme-app"
  data-api-url="/collect.php"
  data-config='{
    "heatmaps": {"enabled": true, "clicks": true, "taps": true}
  }'>
</script>

Heatmap event metadata includes `x_pct`, `y_pct`, `viewport`, `path`, `input`, and sanitized element metadata that respects `datasetAllowlist` and `datasetDenylist`.

SELECT
  json_extract(event_meta, '$.path') AS path,
  json_extract(event_meta, '$.viewport') AS viewport,
  json_extract(event_meta, '$.x_pct') AS x_pct,
  json_extract(event_meta, '$.y_pct') AS y_pct,
  COUNT(*) AS points
FROM events
WHERE app_id = 'acme-app' AND event_type = 'heatmap_point'
GROUP BY path, viewport, x_pct, y_pct
ORDER BY points DESC;

Payload schema

The client POSTs JSON batches. `sendBeacon` may deliver `text/plain`, so collectors should parse the body as JSON regardless of content type.

  "app_id": "acme-app",
  "client_id": "uuid",
  "session_id": "uuid",
  "user": {
    "id": "user_123",
    "account_id": "acct_456",
    "traits": {"plan": "team"}
  },
  "ts": 1781230000,
  "meta": {
    "url": "https://app.example.com/pricing",
    "referrer": "",
    "viewport": {"w": 1366, "h": 768},
    "scroll_pct": 42,
    "focus_ms": 38120,
    "interaction_count": 3,
    "score": 7,
    "user": {
      "id": "user_123",
      "account_id": "acct_456",
      "traits": {"plan": "team"}
    }
  },
  "events": [
    {"id": "e-...", "type": "interaction", "ts": 1781230001, "meta": {"selector": ".cta"}},
    {"id": "e-...", "type": "heatmap_point", "ts": 1781230002, "meta": {"x_pct": 52, "y_pct": 34, "viewport": "1366x768", "path": "/pricing", "input": "click"}}

  ]
}

Collector

`collect.php` is the reference collector. It uses PDO SQLite, WAL mode, soft rate limiting, CORS allowlists, optional API keys, failed payload storage, and a CLI retry worker.

Top-level `user` data is merged into stored client/session metadata. Optional request metadata is available through `$CAPTURE_REQUEST_META`, `$HASH_IP_WITH_SALT`, `$IP_HASH_SALT`, and `$CAPTURE_USER_AGENT`; it is disabled by default.

  • `ux_apps` stores app IDs.
  • `clients` stores stable browser client IDs.
  • `sessions` stores session IDs per client.
  • `events` stores event type, timestamp, and JSON metadata.
  • `surveys` stores survey ID, type, response, and context JSON.
  • `failed_payloads` stores payloads that should be retried by cron.
*/5 * * * * /usr/bin/php /path/to/collect.php --retry >> /path/to/tinyux.log 2>&1

Privacy defaults

tinyux is not a recorder or fingerprinting library. It never captures screen contents, DOM snapshots, keystroke contents, clipboard contents, camera, microphone, console logs, network logs, or form values by default.

Dataset metadata can still identify people if you put sensitive data in `data-*` attributes. Use `datasetAllowlist` for strict deployments.

FAQ

Is this a Hotjar or FullStory replacement?

No. tinyux is for targeted telemetry, contextual feedback, and optional aggregate click/tap heatmaps. It is not a replay, screenshot, or visual behavior suite.

Does it work in SPAs?

Yes. The mutation observer scans new declarative triggers. Call `track()` on route changes if your framework does not reload pages.

What happens when the network drops?

Failed payloads are stored in localStorage with exponential backoff and retried on later flushes.

Can I require an API key?

Yes for normal fetch sends. `sendBeacon` cannot set custom headers, so strict API-key deployments may disable beacon reliance or validate origin/session instead.

Is text survey input safe?

Text surveys can collect personal data because users type free-form content. Use rating, yes/no, and choice surveys when possible.
